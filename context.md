WHAT KIND OF PROJECT THIS IS?
- This project consist in a Music Player as the name suggests. The player will be installed in a windows machine and will be able to reproduce song from files on the computer, create playlist, add images to albuns and song, show lyrics and sincronize then while the music is playing
- The player will only be able do play from local files
- [UPDATED 2026-07-17] Supported formats: MP3, FLAC, M4A, M4B, AAC, WAV, OGG, Opus, WebM (src/main/library/scanner.ts SUPPORTED_EXTENSIONS + src/main/mediaProtocol.ts MIME_TYPES). Limited to what Chromium's <audio> element decodes natively — no transcoding layer exists yet.

ICONS (2026-07-18): per-surface icons supported. Windows derives all icon surfaces from two sources — the exe icon (build/icon.ico: Explorer, Start Menu, desktop, PINNED taskbar) and the window icon (title bar = 16px slot, running taskbar button/Alt-Tab = 24-48px slots). Each ICO size slot can hold different artwork. Optional build/window.ico (gitignore-free, just drop it in) overrides the window icon: dev picks it up directly, packaged builds ship it via build.extraResources → resources/window.ico (missing file = filter matches nothing, build still succeeds, falls back to exe icon). Limitation: a PINNED taskbar button always shows the shortcut's icon (= app icon), not the window icon — Windows doesn't allow overriding that per-window.

PERF PASS (2026-07-18) — four optimizations, all shipped together:
1. Playback ticks no longer re-render the app. currentTime left React state entirely: usePlaybackTime.ts is an external store (useSyncExternalStore) fed by usePlayer's timeupdate/seek via publishPlaybackTime; only NowPlayingBar, FullscreenPlayer, and LyricsView subscribe. PlayerState no longer exposes currentTime — non-subscribing code that needs the position (e.g. seek shortcuts in App) reads getPlaybackTime(). duration stayed as state (changes once per track). Side fix: a new track publishes 0 immediately, so the seek bar snaps to the start instead of holding the old position until the first timeupdate.
2. Scanner parses files through a worker pool (SCAN_CONCURRENCY = 8, shared-cursor while loop). Safe because everything touching shared state (pending buffer, better-sqlite3 calls, counters) is synchronous — only stat/parseFile overlap. getMusicMetadata() now loads once before the loop. Progress IPC throttled to 250 ms (PROGRESS_INTERVAL_MS); the final progress message always fires. Previously each of N files sent one IPC message = one renderer re-render.
3. Missing-file cleanup went async: fs access() checks through the same pool pattern instead of existsSync back to back on the main process (which could freeze IPC for seconds on a waking HDD). Deletes now batch in one transaction. Unreadable-directory protection unchanged.
4. Albums grid: covers get loading="lazy" decoding="async" (opening the tab no longer decodes every cover at once); filteredAlbums and LibraryView's filteredTracks are useMemo'd. Grid virtualization deliberately skipped — lazy images cover the realistic album counts; revisit only if grids reach thousands.
